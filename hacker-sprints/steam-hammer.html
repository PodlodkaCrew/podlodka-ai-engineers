Steam Hammer

github.com/podlodka-ai-club/steam-hammer

CLI-оркестратор, который из короткого GitHub-issue делает готовый fix-branch + Pull Request. Внутри одного бинаря живут две реализации: новое Go-ядро и старый Python-скрипт. Go-обёртка на каждой команде решает, кому отдать управление — это «двухголовый» CLI в середине миграции.

Главная идея

Двухголовый CLI с трекером вместо собственной БД

Снаружи это один бинарь orchestrator, который всегда стартует через Go-обёртку. Но при выполнении каждой команды Go спрашивает себя: «эту задачу я умею в native-режиме или мне нужно дёрнуть старый Python-скрипт?». Если упирается в неподдерживаемый сценарий (Jira, декомпозиция, image-вложения) — явно логирует falling back to python runner и запускает scripts/run_github_issues_to_opencode.py как subprocess. Состояние задачи не лежит в локальной БД — оно живёт прямо в issue/PR-комментариях GitHub как append-only лента маркеров вида <!-- orchestration-state:v1 -->.

Высокоуровневая схема

Как устроен «паровой молот»

Go-диспетчер на каждой команде выбирает мотор: native-flow или Python-fallback. Снаружи — единый CLI, единый state-формат, единый конфиг.

Что важно понять

  • → Один CLI, два мотора. Оператор всегда запускает orchestrator, но за каждой командой стоит выбор: native-Go или fallback в scripts/run_github_issues_to_opencode.py. Граница явная — в stderr летит строка falling back to python runner: <reason>.
  • → Трекер вместо БД. Все переходы FSM пишутся новыми комментариями с маркерами :v1. Recovery бесплатное, история на ладони, параллельные worker'ы координируются через ту же ленту.
  • → Доменное ядро — чистые функции. Решения «какой статус PR», «когда переключаться на review-flow», «можно ли брать задачу» одинаково вызываются из one-shot, daemon и тестов с фейковыми данными.
  • → Безопасные дефолты. Live-daemon требует явного --allow-live-side-effects. Перед каждой git-мутацией — проверка ветки и каталога. Budget- и idle-таймауты kill'ят агента.
Two-headed CLI

Два мотора под одной обёрткой

Проект — в середине миграции «Python → Go». Go-ядро вытягивает функциональность по частям, Python-runner остаётся хранителем легаси-сценариев. Оба видят один и тот же конфиг и одни и те же маркеры в трекере.

Tracker как state-store

Семейство versioned-маркеров

Вместо SQLite или Postgres — append-only лента issue/PR-комментариев. Каждый служебный комментарий начинается с HTML-маркера, к которому навсегда прибит суффикс :v1 — чтобы будущая миграция схемы не сломала чтение старых записей.
