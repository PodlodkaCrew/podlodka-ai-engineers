Night Shift

github.com/podlodka-ai-club/night-shift

Автомат для разбора issue в GitHub Projects: оператор кладёт тикет на доску, и сервис сам ведёт его через три фазы — Specify → Implement → Review — с inline-эскалацией при провале. Под капотом Temporal как движок долгоиграющих workflow и два сменных исполнителя (Codex или Claude) под единым интерфейсом.

Главная идея

Workflow в Temporal — это детерминированная скорлупа фаз, а доска — пульт управления

Workflow знает только, в какой фазе сейчас тикет и куда ему идти дальше. Всё, что трогает время, сеть и файлы — git, GitHub, агенты, проверки — вынесено в activities и переживает падения через heartbeat-чекпоинты. Решение «кого запускать сейчас» принимает не workflow, а отдельный intake-слой, который смотрит на статус карточки и состояние workflow и выдаёт одно из трёх: start / signal / noop. Оператор никогда не «дёргает workflow» — он двигает карточку, а Night Shift сам понимает, что это значит.

Высокоуровневая схема

Как устроена «ночная смена»

Расписание pickup-schedule тикает раз в десять секунд, intake переводит статусы доски в действия, а workflow ведёт тикет по фазам, делегируя всю работу с внешним миром в activities.

Что важно понять

  • → Workflow детерминирован. В скорлупе живёт только switch по currentPhase. Время, сеть и файлы — за её границей, в activities, которые можно ретраить независимо.
  • → Доска — единый UI управления. Оператор двигает карточку, intake переводит движение в нужный сигнал. Правила лежат в одной таблице и тестируются отдельно от Temporal.
  • → Двойной heartbeat. На каждом шаге активность кидает чекпоинт «ответ агента получен», потом «применён». Рестарт между ними не теряет дорогой LLM-вызов.
  • → Один интерфейс к двум провайдерам. AgentSession.run() закрывает и @openai/codex-sdk, и @anthropic-ai/claude-agent-sdk; выбор провайдера склеивается из четырёх слоёв.
Цикл фаз

Четыре фазы — три основные плюс inline-эскалация

Каждая фаза — это связка из phase.ts (что делать), prompt.ts (как просить), response.ts (zod-валидация) и errors.ts. Фазы не знают про Temporal — принимают «мешок зависимостей» и тривиально тестируются.

Диспетчер на проходной

Intake — три возможных действия

Маленький слой, который принимает решение по таблице правил: смотрит, в каком статусе сейчас issue и что говорит уже запущенный workflow про свою блокировку. Эту же логику использует и плановый pickup, и ручной запуск через CLI.

Правило перевода «статус + blockedReason» в действие

Оператор двигает карточку — intake возвращает один из трёх вердиктов. Если ничего не изменилось — noop. Если фаза-кандидат не совпадает с текущей — start. Если workflow ждёт сигнала и условие выполнено — signal.

start
Workflow для тикета ещё не существует
Запустить новый ticket-<N> с нужной стартовой фазой. Из Backlog — со specify; если спецификация уже есть, можно сразу с implement.
signal
Workflow открыт и ждёт оператора
Послать сигнал в существующий workflow — specReviewed, implementRetry, resume и т.п. Workflow проснётся и пойдёт в следующую фазу.
noop
Состояние не изменилось
Ничего не делать. Тикет в нерелевантном статусе, или workflow уже работает над этой же фазой — повторный start вреден.
