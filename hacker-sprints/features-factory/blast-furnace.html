Blast Furnace

github.com/podlodka-ai-club/blast-furnace

Домашняя фабрика для одного GitHub-репозитория. Раз в минуту смотрит issue с лейблом ready, сама ведёт его через девять стадий конвейера и открывает PR — без webhooks, без внешней БД, на одном Codex CLI и Redis-очереди.

Главная идея

Конвейер с одним работником и общим бортовым журналом

Любой переход между стадиями — это запись в журнал, и следующая стадия читает её, чтобы понять, что делать дальше. Состояние не передаётся в payload очереди, оно живёт в файле. Codex играет роль «рук» только на четырёх стадиях из девяти — всё остальное делается обычным детерминированным кодом.

Высокоуровневая схема

Как устроена фабрика

Один процесс на инсталляцию, никаких webhooks и внешней БД — только Redis для очереди и файловая система для журнала прогонов.

Что важно понять

  • → Бизнес-данные не ездят в очереди. В payload только тип стадии, runId, попытка и указатель на запись в журнале. Очередь можно дропнуть и пересоздать.
  • → Workspace в /tmp/. Codex физически не видит ни orchestrator-папку, ни другие прогоны. Не sandbox — но практический файловый забор.
  • → Git и GitHub — у orchestrator'а. Токен только у Blast Furnace. У Codex его нет в принципе, поэтому даже в bypass-режиме он не может ничего залить.
  • → Polling вместо webhooks. Никакого публичного endpoint'а. Цена — латентность реакции в пределах минуты.
Конвейер

Девять станций пайплайна

Issue проходит по жёсткому порядку. Codex участвует только на четырёх станциях (план, разработка, ревью, триаж комментариев). Остальное — обычный TypeScript. Кликните на станцию, чтобы увидеть детали.

Лимиты и инварианты

Безопасные дефолты

Сервис нельзя случайно «оставить молотить вечно». Все таймауты ограничены сверху, ретраи лимитированы для каждой ключевой стадии.

Лимиты ретраев на стадиях

До какого числа попыток фабрика будет крутить ключевые этапы прежде чем остановить прогон или отдать управление человеку.

Распределение работы по типам кода

Из девяти стадий конвейера только четыре полагаются на Codex. Остальные — git, GitHub API, файловые операции, проверка лейблов.

Из чего собран

Тринадцать строительных блоков

От «дежурного по смене» до «контролёра прямо в цехе» — каждый компонент решает одну конкретную задачу. Используйте фильтр, чтобы посмотреть по категориям.
