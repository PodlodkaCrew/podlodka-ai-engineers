Главная идея

Один лонг-таблица состояний + cooperative polling

Вся правда о происходящем лежит в одной Postgres-таблице tasks. Три воркера разносят по ней работу: каждый периодически берёт строку «своего» типа через FOR UPDATE SKIP LOCKED, делает свой шаг и пишет результат обратно. Нет шины событий, нет брокера, нет message queue — только Postgres и три цикла while True: sleep(N).

Вторая большая идея — single handoff contract: между воркерами передаётся не свободный текст, а pydantic-структуры с extra="forbid". Любое решение — что делать дальше, в какой статус двигать тикет, какой комментарий писать — сидит в структурированных полях, а не в summary или details.

Как устроен «тяжёлый подъём»

Один Flask + три воркера + один Postgres. Снаружи — два моста (Linear и GitHub) с mock-двойниками. Между воркерами — только таблица состояний.

Что важно понять

  • → Postgres = очередь + state store. Никакого RabbitMQ/Redis/Celery. Воркеры держат всё состояние в БД и не помнят ничего между циклами.
  • → Кластер задач под одним root_id. Один внешний тикет = много строк tasks: fetch, под ним triage execute, потом impl execute как sibling, deliver'ы, feedback-итерации. Всё связано через parent_id и root_id.
  • → Бэкенд владеет публикацией. Агент по контракту правит только файлы. git commit / git push / gh pr create делает воркер — и только после успешного прогона.
  • → Скрытые блоки в Linear/GitHub. Служебные метаданные живут в HTML-комментариях в description и base64url-footer'е в теле PR. Без отдельной таблицы маппингов.
  • → Local runner — не stub. LocalAgentRunner правдоподобно имитирует триаж и execute, поэтому make demo гоняет полный pipeline без LLM.
Три воркера

Глаза, пресс и почтальон

Каждый воркер живёт в своём цикле, забирает «свой» тип задач из общей таблицы и не знает про существование двух других. Между ними — только pydantic-контракт и Postgres-строки.

Типы задач

Пять видов строк в таблице

В таблице tasks живут пять разных типов строк. Каждый воркер берёт только «свой» тип — это разделяет работу без явного routing'а.
