Spark Gap

github.com/podlodka-ai-club/spark-gap

Робот-надсмотрщик над GitHub Issues. Сам по себе stateless — всё состояние живёт в самом GitHub: один workflow-лейбл и один pinned JSON-комментарий на каждую issue. Раз в минуту тикает, читает мир, дёргает обработчик стадии, записывает обновлённое состояние обратно.

Главная идея

У оркестратора нет ни базы данных, ни локального стейт-файла

Всё состояние живёт в самом GitHub: workflow-лейбл на issue показывает, где сейчас задача в конечном автомате (decomposing, implementing, validating, …), а один pinned-комментарий с маркером <!--orchestrator-state {…}--> хранит JSON с ID сессии агента, номером PR, счётчиками ретраев и watermark-ами. Процесс — это «тиктакающий рефлекс»: раз в минуту прочитал GitHub, дёрнул обработчик по лейблу, записал JSON обратно. Один writer, никаких блокировок, никаких гонок.

Высокоуровневая схема

Как устроен «искровой разряд»

Один Python-процесс, одно poll-петля, одна точка истины — GitHub. Issue обрабатываются строго последовательно, параллелизма нет в принципе.

Что важно понять

  • → Stateless процесс. Падение, рестарт, миграция — ничего не теряется. На следующем тике перечитывается состояние с GitHub и продолжается с того же места.
  • → Никакого параллелизма. Issue обрабатываются строго последовательно. Гонок нет, потому что нет concurrency. Не масштабируется, но это и не цель.
  • → Self-restart на собственном мерже. Если оркестратор смерджил PR со своим же кодом — он замечает это, выходит, обёртка run.sh делает git pull --ff-only и перезапускает Python на новой версии.
  • → HITL — это диалог, а не остановка. На любую парковку (timeout, конфликт, исчерпан бюджет) — постится комментарий с @-mention. Когда оператор отвечает, тот же агент resume-ится с его репликой как followup.
Workflow-лейблы

Десять стадий конечного автомата

На каждой issue ровно один лейбл из десяти — это и есть текущее состояние FSM. Сам лейбл — публичная часть истины; детали (ID сессии, номер PR, watermark-и) лежат в pinned-комментарии с JSON.

Backend lock-on-first-spawn

Два бэкенда, одна issue — один бэкенд

Какой агент первым стартанул на issue, тот за ней и остаётся до конца. Моста для миграции сессии между CLI нет — поэтому бэкенд фиксируется в pinned-state и не меняется.

Lock-on-first-spawn: при первом запуске агента на issue бэкенд (claude или codex) пишется в pinned-state. Все последующие тики используют тот же бэкенд — и тот же session_id для resume.
PR-фидбек — три namespace

Три отдельных watermark-а

У GitHub REST API для PR-фидбека три разных namespace ID комментариев. Если хранить один watermark и применять ко всем — комментарии из одного namespace будут молча пропадать под id-ом из другого. Поэтому в pinned-state живут три отдельных счётчика.

Hardened git

Шесть слоёв брони против собственного агента

Worktree принадлежит агенту — он может посадить там pre-push hook, url.<host>.insteadOf в ~/.gitconfig, fsmonitor-бинарь, credential-helper-скрипт. Поэтому все git push/fetch/merge, которые делает сам оркестратор, проходят через жёсткую обёртку.
