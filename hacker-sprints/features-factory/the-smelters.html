The Smelters

github.com/podlodka-ai-club/the-smelters

Хакатон-оркестратор, превращающий markdown-задачи в закрытые тикеты руками двух агентов — кодера и ревьюера. Очередь живёт в SQLite, роли — это отдельные subprocess'ы, общение — через stdout и append-only JSONL. Внутри живут два независимых оркестратора одновременно — «голый» asyncio и фреймворковый Agno.

Главная идея

Оркестратор сам ничего не знает про LLM

Очередь задач лежит в SQLite, каждая роль — отдельный subprocess, общение идёт через stdout и append-only events.jsonl. Оркестратор лишь дёргает дочерние процессы и читает последнюю строку stdout как JSON-результат. Граница процессов — это и есть граница SDK: подменить backend = подменить модуль. Параллельно живёт второй мир — Agno-workflow, где те же роли собраны не из subprocess'ов, а из step'ов внутри одного процесса. Оба мира делят shared/prompts.py, но в runtime изолированы — сознательное двоежёнство для сравнения подходов.

Высокоуровневая схема

Как устроена «плавильня»

Markdown-задача попадает в SQLite через seed.py. Оркестратор атомарно «клеймит» её, готовит worktree-песочницу и поочерёдно запускает кодера и ревьюера. Каждый агент пишет финальную JSON-строку в stdout — это и есть результат.

tasks/<project>/*.md
markdown-задачи с описанием — единственный вход системы
↓ seed.py идёмпотентно
SQLite (WAL) — очередь
BEGIN IMMEDIATE + claim_next_ready_task — атомарный захват
↓ atomic claim
Orchestrator (asyncio loop)
poll → claim → spawn subprocess → parse last-line-JSON
↓ cwd = worktree, env = config
worktrees/<project>/task-N/
изолированная копия + git init + local.properties для Android
↓ subprocess + stdout drain
coder · reviewer · checker · android_coder
claude-agent-sdk или opencode CLI — JSON-строка в финале
↓ append per line
events.jsonl + tui.py / printer.py
единственная шина наблюдаемости — tail-friendly

Что важно понять

  • → Очередь — это SQLite с одним write-lock'ом на claim. Никаких брокеров, никакого Redis. Атомарность BEGIN IMMEDIATE гарантирует, что два worker'а не возьмут одну задачу.
  • → Граница процессов = граница SDK. Оркестратор не импортирует ни Anthropic, ни Gemini SDK. Контракт с агентом — «запусти команду, прочитай последнюю строку stdout как JSON».
  • → WAL разрешает read-only наблюдателей. tui.py, printer.py, прямой sqlite3 — всё работает параллельно с оркестратором без блокировок.
  • → Recovery — это просто рестарт. На старте reset_stuck_tasks возвращает всех висящих running/review обратно в ready. Падение не теряет работу.
Двоежёнство архитектуры

Два оркестратора в одном репозитории

В the-smelters живут два независимых оркестратора одновременно. Оба читают те же markdown-файлы и шарят системные промпты, но в runtime изолированы — это сознательный приём, чтобы сравнивать «голый» подход с фреймворковым на тех же задачах.

Мир №1 asyncio · daemon
orchestrator.py
«голый» asyncio-loop над SQLite-очередью
  • Daemon в --watch-режиме крутит весь батч задач подряд
  • Каждая роль — отдельный subprocess, результат — последняя строка stdout
  • Атомарный claim через BEGIN IMMEDIATE — всё concurrency без локов в коде
  • На старте reset_stuck_tasks поднимает «висящих» обратно в ready
Когда хорош: производственный батч-режим. Перезапустил упавший оркестратор — подхватил очередь с того же места. Параллельно глядишь состояние через любой sqlite3-клиент.
Мир №2 Agno · interactive
agno_orchestrator.py
интерактивный workflow с live-телеметрией
  • Одна задача за раз с богатой телеметрией: live-таблица, разбивка токенов, cost
  • Две схемы: smelters-mode (Loop) и class-mode (TDD-цепочка)
  • DiffTracker подмешивает diff предыдущей итерации в input кодера
  • Resume-from-crash через свою SQLite — пропускает уже сделанные шаги
Когда хорош: интерактивный запуск одной задачи. Эксперименты с фреймворковым подходом. Пошаговая TDD-разработка одного класса по конвенции <module>/src/{test,main}/kotlin/....
Общая земля: оба мира читают одни markdown-файлы, делят shared/prompts.py и валидаторы, но в runtime никогда не пересекаются. Это позволяет крутить эксперименты «голый asyncio против фреймворка» на тех же задачах без переписывания.
Из чего собран

Двенадцать строительных блоков

От markdown-источника задач до multi-layer safety-инвариантов — каждый компонент решает одну конкретную проблему. Используйте фильтр для просмотра по категориям.

Multi-layer защита

Три рубежа безопасности

Каждый рубеж отдельно может показаться избыточным, но вместе они закрывают разные классы ошибок: «агент полез не туда», «агент сделал опасную команду», «агент написал то, что не собирается».

Рубеж 01 · Containment
Рабочая директория не выходит за worktrees/
Жёсткая проверка через path.resolve(): resolved-путь обязан лежать под worktrees/<project>/task-N/ и не совпадать с оригиналом. Защита от ситуации, когда в БД случайно затёрся путь и агент собрался писать в исходный репозиторий.
Рубеж 02 · Bash deny-листы
Несколько списков запрещённых команд на разных слоях
Кое-где запрещён sudo, кое-где git push, кое-где npm install. У ревьюера ещё и нет Write/Edit в allowed-tools — разделение ролей закреплено разными deny-листами bash-команд.
Рубеж 03 · Compile-gate
Для Android: ./gradlew assembleDebug обязан пройти
Даже если агент сам сказал «всё ок» и RUN_TESTS.sh формально прошёл — неудачный билд срывает задачу. Этот rубеж ловит случаи, когда модель написала синтаксически валидный, но не собирающийся код.
Жизненный цикл задачи

Что происходит с типичной задачей

От markdown-файла в tasks/ до закрытого тикета или провала после трёх попыток — четырнадцать конкретных шагов в custom-режиме плюс альтернативный путь Agno.
