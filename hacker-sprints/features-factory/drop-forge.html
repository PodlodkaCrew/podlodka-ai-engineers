Drop-Forge

github.com/podlodka-ai-club/drop-forge

Go-конвейер, который превращает Linear-задачу в готовый PR на GitHub. Один тикующий мозг раз в N секунд смотрит, какие задачи в каких статусах, и для каждого «активного» статуса запускает соответствующий runner. Linear-статус — конечный автомат, git-trailer — единственный канал передачи состояния между стадиями.

Главная идея

Никакого внутреннего durable state в боте нет

Между стадиями состояние передаётся не через базу или брокер, а через сам Linear (статус + attached PR URL) и через git (producer-trailer прямо в commit message). Упади процесс посреди работы — Linear и git помнят всё, что нужно: следующий тик увидит ту же доску статусов и пойдёт по той же дорожке. Поэтому никакого SQLite, никаких очередей, никаких state-файлов.

Высокоуровневая схема

Как устроена «кузница»

Один компонент-диспетчер (CoreOrch) на каждом тике классифицирует задачи по статусу и для каждой подходящей запускает горутину с нужным runner-ом. Между ними — Linear как source of truth.

Что важно понять

  • → Один синхронный тик. На каждом проходе главный поток собирает задачи, классифицирует и запускает горутины. Нет фоновых очередей, hook-ов, асинхронных подписок.
  • → Linear — source of truth. Внутри бота нет ни БД, ни durable очереди. Перезапуск процесса теряет ровно нулевое количество прогресса.
  • → Producer-trailer в commit message. Информация «кто и какой моделью писал артефакт» живёт прямо в git log. Review-стадия читает её оттуда — не из Linear и не из БД.
  • → Только subprocess + HTTP. Никаких внешних брокеров. Сторонняя библиотека ровно одна — godotenv. Всё остальное — stdlib Go, три внешних CLI (git, gh, codex) и две HTTP-интеграции (Linear, Telegram).
Конвейер

Четыре стадии OpenSpec

Три producer-runner-а «как близнецы» (общий скелет: clone → codex → status → commit → push) и review как четвёртая опциональная стадия. Каждая стадия — отдельный Linear-статус и отдельный пакет в Go-модуле.

Самая сложная стадия

Три опоры кросс-агентского ревью

ReviewRunner — единственная стадия с собственными подпакетами. Она читает commit-историю, форсит закрытую JSON-схему ответа Codex-а и публикует ревью в GitHub идемпотентно через HTML-маркер.

Из чего собран

Одиннадцать строительных блоков

От .env и единого CLI-входа до OpenSpec-каталога самоописательных требований. Используйте фильтр для просмотра по категориям.
