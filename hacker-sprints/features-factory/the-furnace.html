The Furnace

github.com/podlodka-ai-club/the-furnace

Конвейер из Linear-тикета в GitHub PR на Temporal + Docker. Состояние — целиком в Temporal-истории, никакой своей базы. Каждая попытка агентской фазы — отдельный одноразовый docker run --rm контейнер. Три независимых Claude-агента работают по строгой трёхтактной схеме: spec пишет падающие тесты, coder их озеленяет, review судит — но не правит.

Главная идея

Temporal + одноразовый Docker = state-free Furnace

Temporal берёт на себя всё, что обычно требует базы и cron-сервиса: расписание, повторные попытки, хранение состояния, очереди задач. Furnace поверх него — всего лишь набор workflow-сценариев и activity-функций. Агенты живут не в основном процессе, а каждый раз — в свежем Docker-контейнере на одну попытку. Грязный worktree, утечки памяти, побочные эффекты предыдущего раунда — всё это физически невозможно унаследовать, потому что между попытками между ними стоит docker run --rm. Свежий мир на каждую попытку — не свойство, которое поддерживается дисциплиной, а единственный способ, которым попытка вообще может произойти.

Высокоуровневая схема

От Linear-тикета до PR

Оркестратор постоянно крутится поверх Temporal, опрашивает Linear, запускает per-ticket workflow. Тяжёлые фазы агентов уходят на отдельную per-repo очередь и исполняются в одноразовых контейнерах.

Что важно понять

  • → Своей базы нет. Состояние workflow живёт целиком в Temporal-истории. Перезапустил оркестратор — ничего не потерял.
  • → Две очереди — два мира. orchestrator-queue и repo-<slug>-worker разнесены, чтобы долгоживущий оркестратор и одноразовые работяги не конкурировали.
  • → Свой дашборд не нужен. Прогресс смотрят через встроенный Temporal Web UI — там видна история каждой activity и каждого аргумента.
  • → Pause через failure. Когда нужен человек, агент не висит — он создаёт суб-тикет, валит workflow особым типом и оставляет cron-перезапуск разруливать.
Двухконтурная схема очередей

Оркестратор и работяги — по отдельности

Очередей две, и это главное архитектурное разделение. Разная природа задач не мешает друг другу: оркестратор всегда онлайн, а фазы исполняются в эфемерных мирах, ничего не зная друг о друге.

Три агента под одним driver-циклом

Разделение прав как архитектура

Все трое устроены одинаково: системный промпт, набор разрешённых инструментов, набор «терминальных» MCP-инструментов для возврата результата, общий driver-цикл. Различаются только правами на инструменты — и в этом вся идея.

Read-only ревьюер — это архитектурное, а не дисциплинарное решение. Ревьюер, который может «починить сам», теряет независимость суждения. У review-агента нет Edit, Write и нет права запускать тесты — только читать.
Трёхтактный круг

Фазы внутри одного круга perTicketWorkflow

spec → coder → review. Если review сказал approve — тикет в Done. Если changes_requested — следующий круг, причём coder получает выжимку предыдущего ревью. Максимум три круга.
