Главная идея

Workflow — это не скрипт, а DAG

Каждый узел — атомарная штука: «вызови агента с таким промптом», «запусти shell», «спроси человека и подожди». Один универсальный executor умеет крутить такой граф, обрабатывать ошибки, паузы и параллельность. А над ним — отдельный backlog-надзиратель, который превращает GitHub в источник задач для DAG'а: лейблы там — это и есть state machine. Слева — «откуда задачи» (issue, чат, веб). Справа — общая фабрика из executor'а, изоляции, провайдеров и БД. Адаптеры платформ — сменные «носики» сверху.

Высокоуровневая схема

Два слоя над общей фабрикой

Сверху — два независимых источника задач (backlog и DAG-программы), снизу — изоляция, провайдеры и БД событий. Контракты между слоями — узкие.

Что важно понять

  • → Control plane — это GitHub. Лейблы на issue (archon:ready, archon:in-progress, archon:pr-open, archon:done) хранят канонический статус. БД — только bookkeeping и счётчики попыток.
  • → Один executor для всех. Workflow на 3 узла и pipeline с десятком ревьюеров параллельно читаются одним и тем же кодом. Бизнес-логика исполнения собрана в одном месте.
  • → Узкий contract-layer. Workflow-engine не знает про SDK Claude/Codex/Pi — только IAgentProvider. Добавить нового провайдера = одна папка и одна строка в реестре.
  • → Resume по journal'у. События узлов — append-only лента в БД. При resume executor просто пропускает узлы, у которых уже есть node_completed.
Workflow YAML

Семь типов узлов в графе

У каждого узла есть depends_on, опционально when:, trigger_rule, retry, hooks, mcp. Один и тот же executor крутит любой набор узлов ниже как направленный граф.

Адаптеры платформ

Шесть «носиков» снаружи + Web UI

Все они общаются с одним и тем же ядром через единый интерфейс: принять сообщение, привязать к разговору, отдать результат стримом или батчем, проверить, что отправитель в whitelist'е. Workflow-движок не знает, откуда пришёл «start workflow».

Из чего собран

Десять строительных блоков

От «программы агента» в YAML до Web UI — каждый компонент решает одну конкретную задачу. Используйте фильтр для просмотра по категориям.

Configuration pipeline

Четыре слоя настроек, последнее слово — за env

Перед всем этим — «защитный поклон»: первый импорт во всех entry-point'ах чистит .env-переменные, которые Bun автоматически подгрузил из cwd, иначе токены целевого репо протекли бы в harness-процесс.

Жизненный цикл
