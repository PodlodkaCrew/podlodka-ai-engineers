Главная идея

Большая state machine с durable-журналом — и ничего лишнего

Один центральный класс OrchestratorPipeline знает, в какой фазе сейчас задача, и каждый его шаг сначала пишет результат на диск, потом меняет состояние. Поэтому любой шаг можно повторить: он либо увидит «уже сделано» и пропустит, либо доделает. Никакого облака, никаких очередей, никаких оркестраторов — всё крутится на машине разработчика как обычная Python-утилита, а состояние run'а лежит в .gg/ в репо. Процесс можно убить в любой момент и продолжить с того же шага.

Что важно понять

  • → Durable как первичный артефакт. Каждый шаг сначала пишет на диск, потом меняет state. Падение — это нормально: gg resume подхватит run на том же месте.
  • → Параллельные кандидаты без блокировок. N решений рождаются одновременно в разных git-worktree — со своими портами и кэшами пакет-менеджеров.
  • → Sandbox by default. Direct-exec backend'а — opt-in. Если srt-py не установлен, run уходит в Blocked — а не делает работу без защиты.
  • → Никакой LLM-as-a-judge. Победитель выбирается по жёсткой таблице баллов. Decision можно объяснить и воспроизвести.
Deterministic scoring

Жёсткая таблица баллов

Когда кандидаты отработали, их код прогоняется через тесты, линтер, typecheck, security scanner. Дальше CandidateEvaluator выбирает победителя по таблице — никакой LLM в этом не участвует. Идея — чтобы выбор был воспроизводимым и его можно было объяснить одной таблицей.

Final-verification gate

Приёмочный контроль перед публикацией

Прежде чем сделать git push и открыть PR, run проходит четыре отдельные проверки. Если хотя бы один блокер — публикация не происходит, run падает в TerminalFailure с понятной причиной.

Stable fingerprint: каждое замечание имеет стабильный идентификатор. Оператор может пометить его как «знаем, разрешаем» — тогда следующие run'ы не будут спотыкаться об то же место.
