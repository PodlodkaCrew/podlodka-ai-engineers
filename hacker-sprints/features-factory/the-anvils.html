Главная идея

Postgres — единственный источник истины, всё остальное — толстые адаптеры

Всё, что меняет состояние задачи, проходит через одну SQL-транзакцию, которая одновременно делает атомарный UPDATE и пишет аудит-событие. Никаких файлов-state, никаких mutex'ов в Python — гонки между сотней параллельных воркеров разруливает сам Postgres через SKIP LOCKED и optimistic-locking по столбцу version. Над этой базой висит чистое ядро (FSM, scheduler, prompts) под защитой import-linter, а вокруг — толстый слой адаптеров для всего, что трогает внешний мир.

Высокоуровневая схема

Как устроена «наковальня»

Источники задач нормализуются в одну модель и кладутся в Postgres. Поверх — FastAPI control plane с двумя sweep'ами; снизу — два физических тела воркера под одним FSM.

Что важно понять

  • → Один SQL-statement на переход. UPDATE … RETURNING + INSERT INTO events в одной транзакции. Аудит-лог никогда не разъезжается с tasks-таблицей.
  • → Чистое ядро под import-linter'ом. FSM, scheduler, prompts и gates физически не могут импортировать asyncpg или httpx — нарушение ловится сборкой.
  • → Два тела одного FSM. Local-воркер ходит в Postgres напрямую, remote — через HTTP. На удалённой VM не ставится asyncpg; контракт worker-entry-purity защищает это статически.
  • → Deny-by-default для Claude. --disallowedTools Write,Edit,Bash по умолчанию; чтобы «развязать руки», оператор обязан явно поставить WHILLY_AGENT_ALLOW_SHELL=1.
Auto-recovery

Два независимых sweep'а

В лайфспане сервера крутятся два sweep-loop'а — ловят разные типы зависаний. Оба одним SQL-statement'ом, чтобы audit-log не разъехался с tasks-row. Задача никогда не остаётся claimed навсегда.

Defense-in-depth

Шесть независимых слоёв защиты

Никакого «одного большого guard'а» — несколько ортогональных слоёв, каждый закрывает свой класс атак. Если один слой обошли, остальные продолжают работать.

Зрители

Четыре канала наблюдения из одного источника

У базы есть четыре независимых способа смотреть на происходящее, и все они получают данные из одного и того же event-stream'а. Картинка в браузере и в логах не расходится.

Источник — append-only таблица events + Postgres-триггер pg_notify('whilly_events', ...). Один asyncpg-коннект слушает канал и фан-аутит подписчикам.
Из чего собран

Двенадцать строительных блоков

От Postgres-таблиц как источника истины до stop-crane'а singleton'а — каждый компонент закрывает свой класс задач. Используйте фильтр для просмотра по категориям.
