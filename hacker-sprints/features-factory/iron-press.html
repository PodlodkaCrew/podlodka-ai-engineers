Главная идея

Поток управления — это граф, а не LLM-диспетчер

LLM не решает, куда двигаться дальше. Она просто говорит «я закончила хорошо», «я упала» или «нужен человек». Этими тремя статусами помечены рёбра графа, и движок идёт по подходящему ребру. Один и тот же workflow всегда обходится одинаково — нет неожиданных маршрутов. Выходная форма агента зашита JSON-схемой на стороне SDK: модель физически не может вернуть ничего, кроме {status: ...}. Никаких regex'ов, никакого «вытащи json из ответа» — гарантия даётся транспортом.

Высокоуровневая схема

Как устроен «пресс»

От описания пайплайна (TS-фабрика или workflow.json) до файлового журнала run'а, который читает UI. Движок — диспетчер между узлами; узел — либо LLM-сессия, либо скрипт. Они равноправны.

Что важно понять

  • → Граф предсказуем. Закрытый enum из трёх статусов делает обход workflow детерминированным — гонок «куда пойти» нет в принципе.
  • → Движок не знает, кто внутри узла. LLM или git checkout — engine видит один и тот же интерфейс. Добавить новый тип узла — это просто новый класс.
  • → Pass-check — турникет на входе. До execute узел может сказать «уже готово, иди дальше» или «жду человека» — без вызова LLM и без артефактов.
  • → Журнал — файлы. Никаких баз данных. UI просто tail'ит файлы; вернёшься через неделю — увидишь полную картину run'а по архивной папке.
Словарь развилок

Три статуса — и всё

Любой узел возвращает ровно одно из трёх значений. Этими статусами помечены рёбра графа, и движок просто идёт по подходящему. Закрытый enum намеренный: чем меньше вариантов, тем проще нарисовать граф и тем меньше пограничных случаев в engine.

Один интерфейс — две природы

Узлы, которые думают, и узлы, которые делают

AgentNode открывает LLM-сессию, Script-узел дёргает git или GitHub API. Для движка они неотличимы — одинаковый execute(), одинаковый result.json, одинаковая стадия в UI.

Pass-check — общий механизм. Перед execute узел может вернуть статус не делая работу: BA смотрит, есть ли уже AgentImpl-issue в Linear — если есть, сразу Pass. Если на issue висит неотвеченный вопрос — WaitUserInput без всякой LLM-сессии. Идемпотентность и human-gate в одной точке.
Безопасность

Permission profiles — забор вокруг агента

Безопасность стоит не на per-агентских лямбдах, а на нескольких именованных профилях. Профиль указывается строкой во frontmatter узла. Это даёт ровно одно место для security-audit'а вместо обхода десятков узлов.

Два формата — одна семантика

Как описать пайплайн

TS-фабрика для сложных случаев, JSON + markdown — для визуального редактора. Оба формата компилируются в одну и ту же структуру движка.

TypeScript-фабрика для сложных пайплайнов

Функция рядом с кодом узлов, регистрируется в реестре через отложенный импорт (так разрывается циклическая зависимость). Так живут simple и default. Полный контроль, типы, удобно для рефакторинга.

workflow.json + skill.md для drag-and-drop редактора

JSON-описание графа плюс markdown-промпт на каждый узел. Так живут sm, default-json, simple-json. Это и есть то, что генерирует Studio. Сканер находит файлы рядом и докладывает их как ленивые загрузчики.
